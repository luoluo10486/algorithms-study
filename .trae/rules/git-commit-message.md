# Git提交信息生成强制规范，严格逐条遵守1~7条规则
1. 语言约束：仅行首type标识（refactor/docs/feat/fix/test/chore/style）保留英文，**标题、所有明细描述全部使用简体中文**，专业技术名词可保留通用英文术语（如Flowable、BPMN、yaml、Mapper），其余描述语句禁止英文。
2. 标题格式：固定为 `type: 中文一句话概括本次整体改动`，语句通顺完整，无需刻意压缩长度，清晰点明核心变更，标题末尾不加标点符号。
3. 分段排版强制要求：
   - 标题写完后，**空两行**，再输出1、2、3…有序明细条目；
   - 每条明细单独起行，数字+英文点号开头，完整描述一处改动；
   - 不限制条目数量、不限制单条文字长度，改动内容多可自由增加条目。
4. type类型精准匹配规则：
   - feat：新增功能、模块、接口、文件、业务逻辑
   - fix：修复代码bug、边界错误、运行异常、逻辑漏洞
   - refactor：代码重构、框架迁移、依赖替换、结构调整（不新增功能、不修复bug）
   - docs：新增/修改文档、笔记、注释、说明文档
   - test：新增/调整测试用例、测试代码
   - chore：仓库配置、依赖版本、工具脚本、环境文件调整
   - style：仅代码格式、缩进、换行美化，无逻辑变更
5. 明细描述规范：每条条目完整说明「操作+对象+具体内容」，写清修改文件、依赖、数据库、前后端、配置等细节，不用缩写、不省略关键信息。
6. 附加参数：若本次提交关联需求/任务编号，允许在标题末尾追加指令参数（如 scrum -e 工单号 -h:8 -s:30），放在标题文字后方。
7. 禁止行为：不生成纯英文提交、不中英文混杂描述改动、不省略多文件拆分明细、不强制缩短标题字数。
